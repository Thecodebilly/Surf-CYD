## 1. Tide Data Fixes

### Problem
Tides showing all 0's on the display.

### Solution Applied
Added extensive debugging to identify the issue:

1. **NTP Time Sync Check**: The code now verifies that NTP time has synced before making NOAA API requests. If time is not synced (< 1,000,000,000 seconds since epoch), it will log an error.

2. **Enhanced Error Logging**: Each failure point now logs detailed information:
   - WiFi connection status
   - Station ID validity
   - HTTP response codes
   - API response content (first 200 chars)
   - JSON parsing errors with error messages
   - NOAA API error messages if present
   - Number of predictions returned

3. **URL Logging**: The full NOAA API URL is now logged so you can verify the request.

### What to Check

**On your device, check the serial monitor output for:**

- `"NTP time synced successfully"` - Should appear on boot
- `"Fetching NOAA tides for station XXXXXXX from YYYYMMDD to YYYYMMDD"` - Shows station and date range
- `"NOAA API response (first 200 chars): ..."` - Shows what NOAA returned
- `"NOAA returned XX tide predictions"` - Should be ~48 predictions
- `"NOAA tide - Current: X.XX ft (X.XX m), Range: X.XX to X.XX ft"` - Final values

**Common Issues:**

1. **NTP not synced**: If you see `"NTP time not synced yet"`, the device hasn't connected to time servers yet. This should resolve within 5 seconds of WiFi connection.

2. **Wrong date format**: NOAA expects YYYYMMDD format. Since this is 2026, the API might not have predictions that far in the future (NOAA typically provides 1-2 years).

3. **Station not found**: If `findNearestTideStation()` fails, no station ID will be available.

### Expected Tide Values for Mayport (30.3970, -81.4276)

**NOAA Station**: Likely station 8720218 (Mayport, FL)

**Typical tide range**: 4-6 feet (1.2-1.8 meters)

**Note**: Since we're in February 2026, NOAA's prediction API might not have data this far in the future. NOAA typically provides tide predictions for the current year plus one year ahead.

### Debugging Steps

1. Connect to serial monitor (115200 baud)
2. Watch for the log messages above
3. If you see `"Failed to fetch NOAA tide data: HTTP XXX"`, the HTTP error code will tell us what's wrong:
   - 400: Bad request (date format issue?)
   - 404: Station or data not found
   - 500: Server error

4. Copy the NOAA API URL from the logs and try it in a web browser to see NOAA's actual response

## 3. Database Table Creation

### What Changed

**Before**: The API would automatically create the `records` table if it didn't exist.

**Now**: The table **MUST exist** before starting the API server. The server will verify the table exists and exit with an error if it doesn't.

### Why This Is Better

- ESP32 device cannot create tables (security)
- Forces explicit database schema management
- Prevents accidental table modifications
- Follows database best practices

### Creating the Table

**Option 1: Using the provided SQL file**
```bash
cd api
PGPASSWORD=RRqezpsiCoEcZvONMJtOjKmLneThFveJ psql \\
  -h caboose.proxy.rlwy.net \\
  -U postgres \\
  -p 33697 \\
  -d railway \\
  -f create_table.sql
```

**Option 2: Manual SQL**
```sql
CREATE TABLE records (
  id SERIAL PRIMARY KEY,
  name VARCHAR(20) NOT NULL,
  score INTEGER NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_score ON records(score DESC);
```

**Option 3: Railway Web Interface**
1. Go to Railway dashboard
2. Select your PostgreSQL database
3. Open the "Data" tab
4. Run the SQL above in the query editor

## Troubleshooting

### Tides Still Show 0

1. Check serial monitor for specific error messages
2. Verify NTP time synced successfully
3. Try the NOAA API URL manually in a browser
4. Check if station ID is being found (should log station number)

### Database POST Not Working

1. Verify table exists in database
2. Check API server logs for errors
3. Test with curl commands above
4. Ensure `DATABASE_URL` is set in `.env`
5. Verify API server is running

### API Won't Start

1. Check for "DATABASE_URL environment variable is not set" error
2. Verify `.env` file exists in `api/` folder
3. Run `npm install` to ensure dotenv is installed
4. Check database connection string is correct
